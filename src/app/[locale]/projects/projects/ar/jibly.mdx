## نظرة عامة

في مشروعنا مع **رائد الأعمال نزار بلاغة** من **مدينة الجميل**، لم نكتفِ ببناء تطبيق — بل أسّسنا **«جيبلي»** كمنظومة تجارة إلكترونية وتوصيل متكاملة متعددة البائعين، صُمّمت لتخدم السوق المحلي وتُعيد تعريف تجربة البيع والشراء في المدينة.

**جيبلي** تربط العملاء بالمتاجر والسائقين عبر واجهة برمجية موحّدة، وتغطي أكثر من مجال تشغيلي في منصة واحدة: مطاعم، بقالة، صيدليات، طرود، وخدمات إضافية — مع إدارة مركزية كاملة من لوحة التحكم.

## المميزات الرئيسية

**تطبيق العميل (Flutter — iOS / Android / Web)**
- تصفّح المتاجر حسب الموقع والتبديل بين الوحدات (طعام، بقالة، صيدلية، طرود، تاكسي)
- سلة، كوبونات، جدولة التوصيل، وعناوين على الخريطة
- دفع: كاش، محفظة، ودفع إلكتروني عبر **معاملات**
- تتبّع الطلب لحظياً مع خريطة حية للسائق
- محفظة، نقاط ولاء، إحالات، وضيف بدون تسجيل

**تطبيق السائق (Flutter — iOS / Android)**
- استقبال الطلبات وقبولها أو رفضها
- تحديث حالة التوصيل خطوة بخطوة
- تتبّع GPS في الخلفية وبث الموقع للخادم
- لوحة أرباح، سحوبات، ومحفظة

**تطبيق البائع (Flutter — iOS / Android)**
- إدارة المتجر، القائمة، والطلبات الواردة
- طباعة حرارية (ESC/POS) للإيصالات
- تقارير مبيعات، كوبونات، وإدارة السائقين

**لوحة الإدارة وواجهة الـ API (Laravel)**
- إدارة المناطق، الوحدات، البائعين، السائقين، والعملاء
- الطلبات، العمولات، الاشتراكات، والتسويات المالية
- بوابات الدفع (بما فيها **معاملات**)، SMS، وإشعارات Firebase
- تقارير وتحليلات على مستوى المنصة

**جاهز للسوق الليبي**
- دفع إلكتروني عبر **معاملات** (بطاقات نمو، Visa، Mastercard، محافظ)
- SMS OTP للتحقق والإشعارات
- كاش ومحفظة داخل التطبيق
- تتبّع حي للطلبات والسائقين

## التقنيات المستخدمة

- **Flutter 3.38+** — تطبيقات العميل، السائق، والبائع (GetX)
- **Laravel 12** — REST API v1، لوحة الإدارة، Laravel Passport
- **MySQL** — قاعدة البيانات
- **Laravel Reverb / WebSockets** — تحديثات الطلبات والموقع لحظياً
- **Firebase** — FCM للإشعارات، Firebase Auth (تطبيق العميل)
- **Google Maps** — التتبع والملاحة
- **AWS S3** — تخزين الملفات
- **معاملات** — بوابة الدفع المحلية
- **وحدات Laravel** — AI (OpenAI) وTaxModule

## التحديات والدروس المستفادة

بناء منظومة متعددة التطبيقات والوحدات التشغيلية تطلّب **API موحّداً واضحاً** يخدم العميل والسائق والبائع والإدارة من مصدر واحد — أي تأخير أو تعارض في حالة الطلب ينعكس مباشرة على تجربة المستخدم.

**تزامن الطلبات لحظياً** بين تطبيق العميل وتطبيق السائق ولوحة الإدارة كان من أبرز التحديات: WebSockets (Reverb) مع إشعارات Firebase لضمان أن كل طرف يرى نفس الحالة في الوقت المناسب.

**تكامل الخدمات المحلية** — معاملات للدفع وSMS للتحقق — تطلّب التكيّف مع واجهات API المحلية وتدفقات الدفع بالدينار الليبي، وليس فقط «تفعيل بوابة جاهزة».

**بنية متعددة الوحدات** (طعام، بقالة، صيدلية، طرود…) داخل تطبيق عميل واحد علّمتنا أهمية فصل الإعدادات per-zone في لوحة الإدارة، حتى يمكن تشغيل كل وحدة باستقلالية دون إعادة بناء المنصة.
